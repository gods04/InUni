# InUni Password Recovery Design

## Goal

Add a complete email-based password recovery flow to InUni using the existing
Supabase Auth and Brevo SMTP setup. The flow must match the current login page,
remain beginner-friendly, and avoid adding new backend services.

## Scope

This feature includes:

- A `Forgot password?` action in the login form.
- An email-only recovery mode within `/login`.
- A dedicated `/reset-password` page for choosing a new password.
- Supabase recovery-email and password-update integration.
- Loading, success, validation, expired-link, and provider-error states.
- Demo-mode messaging when Supabase is not configured.
- Automated tests for the new behavior and routes.

This feature does not include:

- Changing the existing signup or login design.
- Custom email-template design.
- Administrator password resets.
- Multi-factor authentication.
- Password-strength scoring beyond the existing six-character minimum.

## User Experience

### Requesting a recovery email

The login form displays a text action labelled `Forgot password?` beneath the
password field. Selecting it changes the existing authentication panel into a
recovery form rather than navigating to another page.

The recovery form contains:

- The existing InUni logo and panel layout.
- A `Reset your password` heading.
- A short explanation that a secure reset link will be sent by email.
- One email field.
- A `Send reset link` primary action.
- A `Back to login` secondary action.

After a valid submission, the page displays the same neutral success message
whether or not the address exists:

> If an account exists for that email, a password reset link has been sent.

This prevents account enumeration. Supabase rate-limit and provider errors are
shown using the existing `ErrorState` component.

In demo mode, submitting the recovery form does not simulate an email. It
displays a clear message that password recovery requires Supabase configuration.

### Choosing a new password

The recovery email redirects to `/reset-password`. The page uses the same
single-column panel layout as `/login` and contains:

- A `Choose a new password` heading.
- A new-password field.
- A confirm-password field.
- An `Update password` primary action.

Both fields use `autocomplete="new-password"`. The password must contain at
least six characters, and both values must match.

If Supabase establishes a recovery session, submitting the form updates the
password with `supabase.auth.updateUser`. After a successful update, the app
signs out the temporary recovery session and redirects to `/login` with a
visible success message.

If the email link is expired, malformed, or does not produce a usable recovery
session, the page explains that the link is invalid or expired and provides a
link back to the recovery form.

## Architecture

### Authentication context

`AuthContextValue` gains three focused methods:

- `requestPasswordReset(email)` calls
  `supabase.auth.resetPasswordForEmail`.
- `updatePassword(password)` calls `supabase.auth.updateUser`.
- `signOut()` continues to clear the Supabase or demo session.

The reset redirect URL is generated by the existing redirect helper, extended
to accept a destination path. Signup continues to redirect to `/profile`;
password recovery redirects to `/reset-password`.

The context returns user-facing error strings through the existing `AuthResult`
shape. Demo-mode behavior is handled in the context so pages remain simple.

### Pages and routing

`AuthPage` extends its local mode type from `login | signup` to
`login | signup | recovery`. Existing login and signup behavior remains
unchanged.

A new `ResetPasswordPage` owns password confirmation, validation, update state,
and the invalid-link screen. It is registered as the public
`/reset-password` route because the recovery session is created while the user
arrives from email and must not be blocked by `ProtectedRoute`.

The page determines readiness from Supabase auth state. It accepts a valid
recovery event or an existing temporary session created from the recovery link.
It does not expose raw URL tokens in the UI.

## Data Flow

1. The user selects `Forgot password?` on `/login`.
2. The user submits an email address.
3. InUni calls `resetPasswordForEmail` with
   `https://inuni.co.za/reset-password` as the production redirect.
4. Supabase sends the email through Brevo.
5. The user follows the email link.
6. Supabase exchanges the recovery link for a temporary authenticated session.
7. `/reset-password` validates the two new-password fields.
8. InUni calls `updateUser({ password })`.
9. InUni signs out the temporary session and returns the user to `/login`.

## Error Handling

- Empty recovery email: `Email is required.`
- Recovery request accepted: always show the neutral success message.
- Supabase or SMTP error: show the provider message in `ErrorState`.
- Password shorter than six characters: show a local validation error.
- Passwords do not match: show a local validation error.
- Missing or expired recovery session: show an invalid-link state.
- Password update failure: preserve the form and show the provider message.
- Demo mode: explain that real password recovery needs Supabase.

Buttons are disabled while requests are in progress to prevent duplicate
submissions.

## Testing

Tests will cover:

- Switching from login to recovery mode.
- Rejecting an empty recovery email.
- Sending a recovery request with a trimmed email address.
- Displaying the neutral recovery success message.
- Generating profile and reset-password redirect URLs.
- Rendering the `/reset-password` route.
- Rejecting short and mismatched passwords.
- Updating a valid password.
- Showing an invalid or expired recovery-link state.
- Redirecting to login after a successful update.
- Preserving all existing authentication tests.

The implementation will follow test-driven development: each new behavior gets
a failing test before production code is changed.

## Deployment

No database migration or new environment variable is required. The existing
Supabase project, Brevo SMTP configuration on port `2525`, Cloudflare Pages
deployment, and `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` variables remain
unchanged.

Supabase's allowed redirect URL list must include:

- `https://inuni.co.za/reset-password`
- `https://inuni-uct.pages.dev/reset-password`
- `http://127.0.0.1:5173/reset-password`
- `http://localhost:5173/reset-password`
